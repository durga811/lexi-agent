# Evaluation — How Testing Works in Lexi

> This document explains the **mechanism** of evaluation: how the gold set,
> metrics, and judges fit together, what they can and cannot tell you, and how to
> trust a before/after number when you change the retriever. For the *generated
> results* see [`eval_results.md`](../eval_results.md); for the *gold set itself*
> see [`GOLDEN_TEST_SET.md`](GOLDEN_TEST_SET.md).

## The core idea: you can't measure retrieval without a "right answer"

Every metric works the same way: the agent produces an answer, we extract which
judgments it cited, and we compare that set against a **hand-made answer key**
(the gold set). The whole thing rests on that answer key being trustworthy. So
"are the gold queries correct?" is the right thing to interrogate — **if the gold
set is wrong, every number downstream is meaningless.**

## 1. What the gold set is

`src/eval/gold_set.py` is a Python dict: a handful of queries, and for each, a
hand-labelled list of which `DOC_###` are *supporting* and which are *adverse*.

It is **not** generated by any model or script. A human read the judgments and
decided, *for the specific legal issue in each query*, which cases help vs. hurt.
That is the "ground truth"; the metrics are just set arithmetic against it.

## 2. How a single test run works, end to end

`src/eval/run_eval.py` loops over the gold queries. For each one:

```
1. Drive the real agent:   agent.invoke({"messages": [("user", query)]})
2. Extract citations:      cited_doc_ids(answer)  → {"DOC_001", "DOC_022", ...}
                           (regex: re.findall(r"DOC_\d{3}", answer))
3. Compare to gold:        precision_recall(predicted, gold_all)
```

The agent runs the *full real pipeline* (hybrid retrieval → Gemini → 3-section
answer); the eval scrapes the `DOC_###` tokens out of its prose. **The metric
never inspects the retriever directly** — it judges the agent's *final cited set*.

## 3. The four metrics

**(a) Precision** — *of what the agent cited, how much was actually relevant.*
`precision = |predicted ∩ gold| / |predicted|`

**(b) Recall** — *of what should have been found, how much the agent found.*
`recall = |predicted ∩ gold| / |gold|`

(a) and (b) are **deterministic** — pure set math, no LLM, identical every run.
That is the reproducible "backbone."

**(c) adverse_recall** — *of the dangerous cases, how many did it surface?* Same
recall formula but only over the `adverse` list. Targets the worst failure mode:
silently burying the precedents that cut against the client.

**(d) reasoning / adverse_honesty** — LLM-judged. DeepEval G-Eval asks Gemini (via
the `GeminiJudge` adapter in `judge.py`) to score 0–1 on a rubric. These are
**non-deterministic** — they wobble run to run, and degrade gracefully (skipped)
if `deepeval` isn't installed.

**(e) faithfulness (grounding)** — LLM-judged, the canonical RAG "is the answer
grounded in what was retrieved?" probe. The eval captures the chunks the agent
*actually* retrieved (via a log in `search_corpus`) and passes them as
`retrieval_context`; a G-Eval rubric then checks that every claim is traceable to
a chunk. This is the **confident-hallucination detector** — the dangerous legal
failure (inventing a holding the retrieved text never states).

> **Why a custom G-Eval and not DeepEval's `FaithfulnessMetric`:** the stock metric
> only penalises *contradictions* — a fabricated claim the context doesn't mention
> gets verdict "idk" and counts as faithful. We verified this: a fabricated answer
> scored **1.0** on the stock metric. For law, fabrication *is* the failure mode,
> so the custom rubric penalises any unsupported claim (grounded → 1.0, fabricated
> → 0.0). This is exactly the guidance to "penalise hallucination harder than the
> built-in metric for high-stakes domains."

So the eval is layered: a reproducible deterministic backbone (a, b, c) plus a
softer LLM-judge layer (d, e). It also follows the RAG-eval **retriever/generator
split** — `src/eval/retrieval_eval.py` scores retrieval alone (recall@k), while
faithfulness isolates the *generator* given the retrieved context, so a bad answer
can be traced to the half that broke.

## Alignment with the task objective (does the eval grade the right behaviour?)

The assessment fixes what the eval must track: the agent's **3 required outputs**
(Supporting Precedents · Adverse Precedents · Strategy) and **4 eval dimensions**
(Precision · Recall · Reasoning Quality · Adverse Identification).

**Required dimensions — fully covered**, 1:1:

| Dimension | Metric |
|---|---|
| Precision | `precision` (deterministic, cited doc_ids vs gold) |
| Recall | `recall` (deterministic) |
| Reasoning Quality | `reasoning` G-Eval |
| Adverse Identification | `adverse_recall` (deterministic) + `adverse_honesty` G-Eval |
| (grounding — beyond the brief) | `faithfulness` G-Eval (I9) |

**Agent behaviours — covered, with one gap:**

| Agent must produce | Measured by | Status |
|---|---|---|
| Supporting precedents | precision/recall + reasoning | ✅ |
| Adverse precedents | adverse_recall + adverse_honesty | ✅ |
| **Strategy** (prioritised arguments, realistic compensation range, risks) | — | ❌ **no metric** |
| General queries (minimal research) | `deep=False` queries (Q3/Q5/Q6/Q7) | ✅ |
| Dynamic workflow (no hard-coded steps) | tool-call traces (ADR §3), not a metric | ◐ architecture, by design |
| Ground answers in retrieved text | `faithfulness` (I9) | ✅ |

**Decision & reasoning:**
- **The eval is aligned with the task objective** — all four required dimensions
  are implemented, both deep and general queries are exercised (the flexibility
  requirement), and faithfulness adds the grounding check the brief implies.
- **Known gap: the Strategy section (3rd mandatory output) has no dedicated
  metric.** The `reasoning` rubric ("legally coherent conclusions") brushes it but
  does not assess whether the strategy prioritises arguments, gives a *realistic
  compensation range with a basis in the cited precedents*, and flags key risks.
  **Reason it's a gap, not a defect:** the three deterministic + three judge
  metrics cover *which* precedents and *how grounded/honest* the answer is; strategy
  quality is a distinct, subjective axis that needs its own G-Eval rubric.
- **Decision: defer a "Strategy Quality" G-Eval to a follow-up** rather than block
  the faithfulness baseline. When added it slots in exactly like the other G-Evals
  (criteria + `[INPUT, ACTUAL_OUTPUT]`, gated on `deep` queries).
- **Dynamic-workflow is intentionally not a metric** — the assessment frames it as
  an architecture property; it's evidenced by the tool-call-count contrast between
  simple and deep queries (ADR §3), not a score.

## 4. Is the gold set "correct"? — strengths and limits

**What makes it trustworthy:**
- It is **documented and reasoned**, not guessed. The labelling methodology is
  recorded: read each judgment's parties/facts/holding, classify by theme, label
  support vs. adverse *for that query's specific legal issue*.
- It encodes **deliberate judgment calls** (e.g. excluding criminal
  unlicensed-driving convictions from an insurer-liability query to protect
  precision; de-duplicating near-identical judgments).

**What makes it limited — keep this in mind before A/B testing:**
1. **It is small.** With ~15 gold docs on a query, one citation flipping changes
   recall by ~0.067. **A delta smaller than ~0.05–0.07 is within noise.** This
   ruler detects a *reranker-sized* jump, not a 1% tweak.
2. **It is one annotator, unaudited.** No second labeller, no inter-annotator
   agreement. A mislabelled borderline case is baked into every score silently.
   "Correct" means "one informed reading, defensibly documented," not "ground
   truth from a higher authority."
3. **It judges the agent's cited set, not the retriever.** A recall miss could be
   the retriever (never surfaced the doc) *or* the agent (surfaced but didn't
   cite). The current harness **cannot tell these apart**.

## 5. Why this matters before implementing improvements

- **Trust the backbone, not the judge, for A/B.** Precision/recall/adverse_recall
  are deterministic, so a before/after delta above the noise floor is real signal.
  The G-Eval scores wobble; don't claim "+0.05 reasoning" from a single run pair.
- **To attribute a delta to *retrieval* specifically**, add a **retriever-level
  metric** the current harness lacks — RAGAS-style ContextRecall/ContextPrecision:
  run each gold query straight through `get_retriever()` and check whether the
  gold docs appear in the retrieved chunks, *before* the agent reasons. That
  isolates the retriever from the LLM.

## 6. The measured-improvement discipline

A/B one variable at a time:
1. **Baseline** is captured in `eval_results.md`.
2. **Change one thing** (reranker, ensemble weights, counter-query tool).
3. **Re-run** `PYTHONPATH=. uv run python -m src.eval.run_eval`.
4. **Diff the two reports** → that delta is your evidence.

The ADR sentence to aim for — *"hybrid added +X recall, reranking added +Y
precision, the counter-query step took adverse-recall from A to B"* — is not a
feature list. It is a **measured causal chain**: each design choice attached to a
number it moved. The placeholders are filled by running the eval before and after
each change. "Reason about alternatives" is the grading signal: considered
options, picked one, **proved with a metric** it was right.
